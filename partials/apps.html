Creating Apps

Apps are local JavaScript and CSS files that run as asynchronous packages.

An app will have at least two to three parts:

  1. A directory under assets/apps/
  2. A main.coffee file
  3. Optional css or js resources main.js will run.

Hello World app

Let's create the most basic of apps. This will display a javascript message that says "Hello World".

  1. In apps, create a new directory called "helloworld"
    •  helloworld
  2. In helloworld/, add a blank file called main.coffee
    •  helloworld
      •  main.coffee
  3. Add the following to main.coffee 
    alert 'hello world'
  4. In settings.coffee, add 'ks:helloworld' to your k$.apps array. 
    apps: [
  'ks:sample-app'
  'ks:tinygrowl'
  'ks:ang-app'
  'ks:helloworld'
]
  5. Now when you refresh your page, you should see the alert. Keep reading for more complex use cases.

Hello world with extra resources

This app will do the same as our first but will get the alert from another js file and load a css file.

  1. Create the app from above
  2. Erase main.coffee and add the following 
    define [
 './helloworld'
 './helloworld.css!'
], () ->

    Be sure you have the "!" after the .css

    We're telling JSPM to look for two resources, helloworld.js and helloworld.css. We don't have to write '.js' but you'll need to specify a css file is being loaded with the extension and an exclamation point.

  3. Create a blank helloworld.coffee in /apps/helloworld/ and write the following 
    alert 'hello world'
  4. Create a blank helloworld.styl in /apps/helloworld/ and write the following 
    body
  background-color: orange !important

    This is a stylus file, which will be converted by roots to a .css file as we watch and compile. The !important suffix overrides any other declaration of background-color on the body. This won't be pretty, but it will be a useful test.

  5. Refresh the page. You should now see both the "hello world" alert and a yellow page. Keep reading, there's more we can do with apps.

Hello world with dependencies

Continuing with our hello world app from the previous two examples, this app will need to know jQuery is loaded before it tries to use $ functions.

  1. Create the app from above
  2. Edit main.coffee so it reads as such: 
    define [
 'jquery'
 './helloworld.css!'
], ($) ->
  alert 'Hello world. Running jQuery version ' + $.fn.jquery
  3. Delete helloworld.coffee. We no longer need this.
  4. Refresh. You should now see what we had before, but with the jQuery version in the message. This proves the app is waiting for jQuery to be loaded before executing any script.

Hello world with a chain of dependencies

In more realistic scenario, we may be writing code that relies on a library which relies on jQuery. JSPM's Require.js syntax provides a very clean, DRY way to handle this.

Let's say I want to use garlic.js. This caches all your form values in the user's browser in case they accidentally navigate away.

  1. Add the directory "garlic" to /apps
    •  garlic
  2. Create a blank "main.coffee"
    •  garlic
      •  main.coffee
  3. Add the following to main.coffee 
    define ['garlic'], () -> $('form').garlic()

    Notice we didn't denote the local directory with ./. This is because we're going to define it later in _jspm.coffee.

  4. Open _jspm.coffee (in the js/ directory)
  5. Add the following to map: 
    'garlic': 'cdnjs:garlic.js/1.2.2/garlic.min'

    This tells JSPM garlic means a CDN link to garlic on cdnjs.com. Because garlic.js is hosted on cdnjs.com, we can use JSPM's shortcut syntax.

  6. Add the following to shim: 
    'ks:garlic': ['jquery@2.0']

    This will ensure garlic.js is only allowed to run if jquery has been loaded as well. By creating this requirement in _jspm.coffee, we also ensure this is a rule throughout the application.
